RustEmet1987.txt

Replicating Rust 1987 using DDP.

This explains how the Ox program RustEmet1987.ox replicates the model displayed in Table IX, Column 2, Row 1 and 2 model in Rust Econometrica 1987, the celebrated bus engine replacement model. The program puts out computed replacement probabilities that match up with those shown in Figure 3 on page 1025 of the paper.

The code segments show below are not exactly the same as in the Ox file, which includes additional code beyond the basics shown here.

The Model

The estimated model can be summarized as follows: 
Decision Horizon:    T = ∞
Action Variables (α):  i ∈ {0,1}
State Variables (θ):   x ∈ 0, ... , 89.
    x' = (1-i)x    w/ prob. θ31
         (1-i)x +1 w/ prob. θ32
         (1-i)x +2 w/ prob. 1-θ31-θ32
If x = 88 or 89 then the increment probabilities are changed accordingly.
Utility:    U = -[iRC+(1-i)θ1mx] + n     (5.1) p. 1015
Parameters:
    Row 1
        δ = 0.9999;
        RC = 10.07
        θ1 = 2.293
        θ3 = < 0.3919,0.5953>
    Row 2
        δ = 0;
        RC = 7.6538
        θ1 = 71.5133
        θ3 = < 0.3919,.5953>
Notes:
  • Each value of x corresponds to 5000 mile interval of odometer readings. The article seems to suggest that x actually enters the code in terms of thousands of miles (0 to 445,000), but I was not able to replicate results based on that. I was able to replicate replacement probabilities only when x takes on values between 0 and 89.

  • The parameter RC is the bus engine replacement cost. θ1 is the operating cost given mileage x. The parameter m=0.001 is a scaling factor, which appears to rescale odometer readings. However, results were replicated when x is the bin number (0 to 89) not the odometer category.

  • The paper refers to a normalization when displaying V() (see Figure 2, p. 1025). The normalization is not explained, but my calculations seem to explain why it is necessary. The closed form extreme-value value function iteration involves taking exponents of negative values (operating costs). When δ=0.9999 and n=0 above the result is numerical overflow in exp() With n>0 set properly the arguments are kept closer to 0. In the replication the normalization n is set to the maintenance cost for x=89/2, the median mileage category.

    • Implementation in DDP

    To Implement this Model in DDP, the problem is derived from the Ergodic class. 
    class Zurcher : Ergodic	{ 	
    enum{disc,RC,theta1,theta3,Nparams}; 	
    static const decl NX    =   90;	
    static const decl mfact = 	0.001;	
    static const decl pars 	= {{ 0.9999,10.07 ,2.293 , <0.3919,0.5953,1-0.3919-0.5953> },
                               {     0.0,7.6538,71.5133,<0.3919,.5953,1-0.3919-0.5953> }};
    static decl row; 	
    static decl i; 	
    static decl x; 	
    static decl normalization; 	
    static Run(); 	
    static Reachable(); 			
           Utility(); 	
}

    The Zurcher class defines the constants referred to above. It also declares static members for i and x. For convenience it uses a static method to set up and run the model. The two required methods are the static Reachable(), which can be named whatever the user likes and non-static Utility().

    The file RustEmet1987.ox is a standalone program, so it includes a main() routine, which can simply call the run-and-simulate routine sending the row to replicate (the first row is 0, the second is 1): 
     main()	{ 	Zurcher::Run(); 	}

    Two elements of the implementation are short and simple: 

    Zurcher::Reachable()	{ return new Zurcher(); }
Zurcher::Utility()  {
    return   -(A[Aind]*pars[row][RC] + (1-A[Aind])*pars[row][theta1]*mfact*x.v) +normalization; 	
    }

    The first method is called by DDP while it generates the state space Θ Each time it is called Reachable returns a new instance of the Zurcher class, which will be a another endogenous point θ. If the state space had states that were infeasible to reach then Reachable would look at the current values of the state variables (in this case x) and return an integer if the point is not feasible. This feature allows the code to avoid unnecessary visits and reduce memory requirements in very large models. Note that Reachable can be called anything and it does not need to be a method. It could be a standalone function. However, if it is a method of Zurcher then it must be declared static.

    The second routine must be named Utility and it cannot be static. The form of the expression returned as the value should look related to the mathematical expression U() above. First, note that the parameters are accessed as elements of the array pars. Next, the current value of x is accessed as x.v. Equivalently you can access current values as RV(x).

    Finally, Utility must return a matrix of values for each exogenous value and each action vector. In this case the action variable is simply the variable i and there are no exogenous state variables. Further, DDP allows for different sets of feasible actions depending on the state. But again in this model engine replacement is always feasible. In mathematical notation, we would write the choice set as α ∈ A(θ). This translates to A[Aind], which is the matrix of feasible actions at the current state. A is a static array of the distinct feasible sets. Aind is the index into the array specific to the current state θ. Since Zurcher's problem has a single feasible set, A has only one matrix and Aind = 0 at all states.

    Nearly all the original programming is in the other method declared in the Zurcher class. 

    Zurcher::Run()	{
    decl chprob; 	
    Ergodic::Initialize(); 	
    AddActions(i = new ActionVariable("i",2)); 	
    AddEndogenousStates(x = new Renewal("x",NX,i,pars[0][theta3]) ); 	
    CreateSpaces(Zurcher::Reachable); 	
    chprob = <>; 	
    for (row=0;row<sizeof(pars);++row) { 		
        SetDelta(pars[row][disc]); 		
        normalization = pars[row][theta1]*mfact*NX/2.0;
        Vsolve(ExAnteExtremeValue); 		
        Vprint();					 					
        chprob |= reverser(ChProbMatrix[1][]); 		
        } 	
    Draw(0,chprob); 	
    SaveDrawWindow("Zurcher-Figure3-Replication.pdf"); 	
    }

    The user's program must be written to do these tasks in order:

    • Call the method Initialize for the parent class. Since Zurcher is an ergodic problem the program calls Ergodic::Initialize().
    • Create all the state and action variables in the model and add them to the model.
    • Call the method CreateSpaces(); and send the function that returns either an integer or a new instance of the class (in this case Zurcher::Reachable.
    • Set the value of key parameters, notably the discount factor δ. (This can be done at any time before using the model.)

    After these tasks are completed the model can be solved and used.

    AddActions(i = new ActionVariable("i",2)) does two things. It first creates a new (binary) action variable (an element of α) and stores it in the variable i. Then this variable is added to the model.

    The next line creates the state variable x and adds it to the model as an endogenous state (an element of θ). The variable is of type Renewal, which captures exactly the process in Rust. To properly capture the transition it must be coordinated with the choice variable i. To do this i is passed to Renewal() as an argument, along with how many categories x takes on and the vector of increment probabilities. The length of the vector determines how different states are possible next period.

    The code inside the for() loop solves the model twice, for the δ=0.0 parameters and for δ=0.9999. The code can be fairly simple because the transition probabilities for x are the same in the two cases. If they differed across rows they would have to be sent to DDP differently. For each set of parameters the value function is solved through Bellman iteration by Vsolve(). It prints out the results and captures the probability of engine replacement as a row vector. Then the choice probabilities are graphed to be compared with Figure 3 in the paper.

    Readers familiar with Rust-styled model will notice that so far there has been no mention of the error terms in the utility, denoted ε in the paper. In the Zurcher model these errors are not only additive and IID across actions and states, but they also have no effect on future states. In the DDP framework these types of errors are considered a smoothing method which affects how the model is solved, but state variables do not have to be created for these terms. This is a major contrast with exogenous states in DDP, which are also denoted ε. They too are IID across states and time, but can help determine the transition of endogenous states even after controlling for actions. For example, in a job search model the wage offer might be IID, and if accepted it enters the state next period (perhaps because of on-the-job search). This kind of variable would not satisfy Rust's Conditional Independence property. But in DDP the wage offer is an exogenous state variable. Separating exogenous state variables from endogenous states saves memory and calculations. In many models, terms that satisfy conditional independence are smoothing out choice probabilities.

    The smoothing method that matches the method in Rust 1987 is ExAnteExtremeValue. When sent to Vsolve() this ensures that the closed forms for EV exploited in the paper are used during Bellman iteration, equations (4.13) and (4.14) on page 1012.

    The values of EV do not match those shown in Figure 2 of the paper. This is likely due to Rust using a different normalization than the value used here. However, the choice probabilities do appear to match very well those shown if Figure 3, as the image below suggests.

    Two issues arise. First, the paper labels the x axis as thousands of miles since engine replacement. This means that x should take on values 0, 5, ..., 445. It also suggests that the factor on costs scales 300,000 to 300. However, using this scaling I was not able to replicate the choice probabilities. Only when x takes on values 0, ..., 89 and the scaling factor still equals 0.001 was I able to reproduce the choice probabilities.

    Another use of the model is to simulate data using Simulate(). The output file shows a simulated panel of buses, including the months and mileage at every replacement. The simulate routine returns a panel of observations. But in the case of this model it is sufficient to show only months when an engine is replaced, the age of the bus when it occurs and the mileage reading. So the replication program deletes simulated months when no replacement occurred. It also appends the row number for the model.

    For example, the first simulated bus data looks like this

    
Simulated data
   row   bus monthst.ind Const     x     t    t'     i
     0     0   116    72     0    72     0     0     1
     0     0   175    36     0    36     0     0     1
     0     0   269    52     0    52     0     0     1
     0     0   350    51     0    51     0     0     1
    Bus 0 had four replacements, which occurred in months 116, 175, 269 and 350 of its lifetime. The mileage on the odometer at replacement was in the interval [5000x,5000(x+1)]. Note that the column labeled Const is a placeholder for exogenous state variables. If the model has no such variables then DDP inserts a constant there. This makes it possible to construct indices into the state space without requiring a check of zero dimensions. The other state variables listed, t and t', relate to time. Since the model is stationary there is no separate time variable. If this were a finite horizon or non-stationary environment then t would contain the age of the process. DDP allows for random aging so t would not necessarily be equivalent to the months column. The t' variable is used during iteration on the value function. During simulation it is not used and would always have the value 0.
