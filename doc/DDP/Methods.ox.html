 Methods.ox

Different ways to solve the dynamic program.  ⇩ 

  1. Overview

    2. Objects derived from the Method class are algorithms that solve the user's DP problem. A user's program can create more than one method object and apply each to the problem.

    Every Method has a Solve() function that applies the method to the DP problem.

    The method can be sent as an argument to Data or other objects so that the solution can be nested within some other process.

    Example: after setting up the DP problem apply value function iteration to the problem:
    ⋮
CreateSpaces();
decl vmax = new ValueIteration();
vmax.Volume = LOUD;               //V(θ) printed when computed
vmax -> Solve();
⋮
    Methods are General
    Unless the method relies on a specific clock (e.g. ERGODIC) the method will properly iterate on whatever clock the model uses. For stationary models (or for stationary phases of a model) the fixed point will be found. For finite horizon models/phases, backward induction will be used.
    Two or more solution objects can be created and applied to the model for comparison purposes.
    Solution methods work at the highest level of the DP environment.
    To carry out the work they call methods to handle fixed and random effects.

    What happens with a call to Solve()
     
     
    Solve
     
    		 Use a method to solve Bellman's equation for all DPs by calling the FixedSolve() task.
     
     
     
    FixedSolve
     
    		 Loop over fixed effect variables in γ, solve each problem by calling the RandomSolve() task.
    Reuse the solution storage space for each fixed effect to conserve on memory.
     
     
    		  
     
    RandomSolve
     
    		 For the current fixed group, loop over random effect variables in γ. Solve each problem by calling Method::GSolve().
    Reuse value iteration space but store choice probabilities for all random effect groups.
     
     
    		    
     
    Method::GSolve
     
    		 Given the fully specified fixed vector γ, which defines an element of the group space, solve the DP problem.
     
     
    		    
     
    Inner loop of GSolve
     
    		 
        0. EndogUtil         initialize over endogenous states θ
    1. Run            loop over bellman iterations (or other task)
    2. ValueIteration::Update         check convergence/work backwards (return to 1 until finished.)

    Hooks
    The user can insert static functions at different points in this process using Add().
    HookTimes are the integer codes for when the function should be called.
    For example, to say hello after the inner loop above is done i.e. GSolve() ends, do this:
     sayhello() { print("hello"); }
 ⋮
 Hooks::Add(PostGSolve,sayhello);

  2. Value Function Iteration Methods

    3. Value function iteration applies Bellman's EMax operator, \(V(\theta)\ =\ \max U() + \delta EV(\theta') \) to solve for the value function and optimal choice probabilities.

    1. Brute Force Iteration
      2. ValueIteration performs brute force Bellman Equation iteration.
      Example
      CreateSpaces();
⋮
decl vmax = new ValueIteration();
vmax -> Solve();
    2. Keane-Wolpin Approximation
      3. KeaneWolpin is derived from ValueIteration and approximates the solution to the value function at a subsample of the state space.
      Example
      CreateSpaces();
⋮
SubSampleStates(0.4);
decl kw = new KeaneWolpin();
kw -> Solve();

  3. Conditional Choice Probability Methods

    4. CCP methods use estimates of conditional choice probabilities and inverts a function of them to compute (differences in) the value function without Bellman Iteration. They rely on the CCP task which smooths data on observed choices to produce an estimate of Ρ*(α;θ) that does require a solution to V(θ).

    1. Hotz-Miller
      2. HotzMiller is the basic CCP method that inverts a function of the estimated CCPs to compute (differences in) the value function without Bellman Iteration.
      Example
      CreateSpaces();
⋮
HM = new HotzMiller(data);

    2. Aguirregabiria and Mira
      3. AguirregabiriaMira is derived from HotzMiller. It iterates on the Hotz-Miller inverstion to gain efficiency in estimates

  4. Hybrid Solution Methods
    5. Some methods for solving DDP rely not on non-linear optimization and root-finding. These methods combine DDP and FiveO.
    They are categorized as Hybrid Methods and discussed separately.

 KeaneWolpin : ValueIteration : Method : FETask : GroupTask : Task : DP

Approximate EV from a subsample of Θ using Keane-Wolpin (1994).

KW Approximation computes complete "brute force" max v(α) operator on only a randomly chosen subsample of points in the endogenous state space Θ. The results at these points are used to predict (extrapolate) choice probabilities at the non-sampled states without max v(α) computations.

KW is useful when the endogenous state space Θ and the fully exogenous state space are large.

Recall that the exogenous vector ε contains discrete states that are IID and whose values do not affect the transition of other variables directly (only indirectly through the choice α).
Semi-endogenous states, η, are not supported in this method.
A fatal error is produced if they appear in the model when a new KeaneWolpin object is created.
Different feasible sets A(θ) are allowed, but ...
The feasible set must be the same size at each state at a give clock setting t.
A warning message about this is issued if more than one feasible set exists.
When the dimension of the exogenous space is large KW can accomplish two things:
  1. Drastically reduce the computational cost of value function iteration. This is because applying the max operator to each value of ε at a non-sampled θ is replaced by applying it to only one value of ε
  2. Drastically reduce the storage required by the model. This is because utilities and choice probabilities are only stored for the single value of ε at non-sampled states
KW Approximation works well if The extrapolation method is good at approximating the value function at non-sampled states for a relatively small number of sampled states.

At non-sampled endogenous states the one exogenous vector for which max operator is applied is the median/mean ε That is:
Assuming that each element of ε is mean zero and symmetric and the discrete points also map into symmetric actual values, then the median discrete value corresponds to the mean and median value of the ε vector
For example, if each element of ε takes on 5 values (0…4) then the 2 value is the median and will be used for the extrapolation.
For this reason, it makes sense to have exogenous state variables take on an odd number of values when using KW approximation in DDP.

The key elements of the approximation

  1. The points in Θ to subsample at each clock setting, I::t.
    • The sampling is controlled by SubSampleStates(), which takes 3 optional arguments. See its documentation for an explanation.
    • At the subsampled points in θ all the exogenous states are iterated over to compute the full value function.
    • This value of the value is stored as the explained value for the approximation.
    • The explanatory values are also stored, in the default, the choice-specific values at the MEDIAN point in the exogenous vector ε and the max of them.
  2. The type of approximation used
    • Keane and Wolpin's preferred approach is to run a regression at the sampled states
    • The regression is run to explain the value at sampled points then applied to non-sampled states to predict the value.
  3. The Specification of the approximation
    • KW's preferred specification is the default, but it can be replaced by the user (no help yet available on this).
    • The default is to run a linear regression in the V-v(α) vector and the square root of the vector

Details

Brute force value iteration with exogenous and endogenous state variables can be written
Emax(θ) ≡  ∑ ε Ρε V(ε,θ)
which is the expected value of the maximum 
V(ε,θ) = max α∈θ.A  v(α;ε,θ)
When ε and θ both include many states, visiting every value of ε for every value of θ can be expensive.
KW Approximation computes Emax for a randomly selected subset of states at a given t:
ΘKW(t) ⊂ Θ(t).
where Θ(t) is the subset of Θ with states at time t.
Then it interpolates Emax with a function (such as a linear regression) of the values of v(α;ε,θ) at a single exogenous vector, ε, including the non-linear transformation: 
maxE(θ) ≡ max α∈A(θ)  v(α;ε,θ)
When ε = E(ε) then this is indeed the max at the expected exogenous shock.

Then, for points θ ∉ ΘKW(t) it extrapolates the value based on only maxE(θ) and v(α;ε,θ).
.

KW (1994) report the extrapolation is sufficiently accurate in their model so that the simulation bias in parmeter estimates based on it is quite small.
The built-in components of KeaneWolpin use the KW (1994) "linear and square root" regression specification.
However, these components can be replaced by specifications or interpolating functions of the user's choice. To do this, derive a new class from KeaneWolpin and substitute for the virtual components.

The amount of computations saved is easily approximated. Let ε.D, Θ.D and ΘKW.D denote the cardinality of the sets (following the notations used elsewhere). Then the proportion of total max() operations avoided is 
(ε.D-1)(Θ.D-ΘKW.D) / Θ.D

VISolve ( ToScreen , aM , MaxChoiceIndex , TrimTerminals , TrimZeroChoice )
Simplified Value Iteration model solution.
Parameters:
ToScreen TRUE [default] means output is displayed .
aM address to return matrix
0, do not save [default]
MaxChoiceIndex FALSE = print choice probability vector [default]
TRUE = only print index of choice with max probability. Useful when the full action matrix is very large.
TrimTerminals FALSE [default]
TRUE means states marked IsTerminal are deleted
TrimZeroChoice FALSE [default]
TRUE means states with no choice are deleted
Returns:
TRUE if method fails, FALSE if it succees
Note: All parameters are optional, so VISolve() works.
This function
Creates a ValueIteration method
Calls DPDeubg::outAllV()
Calls Solve()
deletes the solution method

This routine simplifies basic solving. Simply call it after calling CreateSpaces(). Its useful for debugging and demonstration purposes because the user's code does not need to create the solution method object and call solve.

This would be inefficient to use in any context when a solution method is applied repeatedly.
